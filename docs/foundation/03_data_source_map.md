# Data Source Map

This document maps the flow of data within the Golf VX Marketing Dashboard, detailing where data originates, where it is processed, and where it is ultimately stored. Understanding this map is critical for preventing data corruption and ensuring accurate reporting.

## Primary Storage: MySQL / TiDB

The MySQL database serves as the ultimate source of truth for the application. It stores all persistent data generated by the application itself or ingested from external sources.

The database schema, managed by Drizzle ORM, defines tables for users, campaigns, marketing channels, members, revenue records, tasks, and form submissions. Any data that needs to survive a server restart or container redeployment must be stored in this database.

## External Data Ingestion

The application continuously ingests data from a variety of external platforms to provide a comprehensive marketing overview.

*   **Meta Ads:** The system fetches campaign performance metrics, ad spend, and audience insights from the Meta Ads API. This data is temporarily cached in local JSON files to respect API rate limits before being processed and displayed on the dashboard.
*   **Acuity Scheduling:** Appointment bookings, trial sessions, and customer information are pulled from the Acuity API to track conversions and revenue.
*   **Toast POS:** Daily transaction summaries and revenue data are synced from Toast to provide accurate financial reporting.
*   **ClickFunnels:** Funnel performance metrics and form submissions are ingested via the ClickFunnels API.
*   **Boomerang POS:** Member data and membership status are synced from Boomerang.
*   **Encharge:** Email marketing metrics, broadcast performance, and subscriber segments are pulled from the Encharge API.

## Data Processing and Caching

The application utilizes a local file cache specifically for Meta Ads data. This cache, located at `/tmp/golf-vx-meta-ads-insights.json` and `.meta-ads-cache/insights.json`, is managed by background scripts.

This local JSON cache is explicitly not a source of truth. It is an ephemeral optimization layer. The data within this cache is periodically refreshed from the Meta Ads API. If the cache is deleted or lost during a container restart, the application will automatically rebuild it during the next scheduled synchronization cycle.

## Legacy Data Sources

The application maintains a legacy synchronization process with Google Sheets, primarily for Annual Giveaway applications.

Historically, form submissions were appended directly to a Google Sheet. The current architecture writes these submissions directly to the MySQL database. However, a background job still exists to sync data from the Google Sheet to ensure no legacy entries are missed. This Google Sheets sync is considered fragile and should be treated as a secondary, fallback data source rather than the primary ingestion method.
